## 2. 架构与决策

### 2.1 核心决策
1. **E2E 测试走进程内 dispatch**：不启动真实 worker 子进程，直接 `app.run_command(env)` 走 `dispatch`。理由：CI windows-latest 无 sidecar 环境；进程内可复用 `in_memory` DB + fake providers，快速且确定性高。真实子进程集成测试推 V0.2。
2. **项目导出/导入用 zip**：`ExportProject` 把单个 `content_project` 的所有 `content_versions` + `source_assets` 元数据（不含二进制源文件，MVP 只导元数据）打成 zip（含 `manifest.json` + `project.json` + `versions.json` + `assets.json`）。`ImportProject` 从 zip 恢复，重映射 id 避免冲突。理由：MVP 验收"5 个项目数据一致"只需元数据；二进制源文件导出推 V0.2。
3. **备份恢复走文件拷贝**：`BackupWorkspace` 显式触发 `shutil.copy2(stepwork.db, backups/stepwork-<ts>.db)`，返路径。`RestoreWorkspace` 从指定备份恢复（关闭当前连接 → 覆盖 → 重连）。复用 `bootstrap._resolve_stepwork_home` 路径约定。
4. **日志落盘用 RotatingFileHandler**：`$STEPWORK_HOME/logs/worker.log`，单文件 5MB × 3 份轮转。失败时降级到 stderr 不阻塞启动（W8_PLAN D5 P1 后置，W9 兑现）。复用 `_mask_secrets` 在写入前过滤密钥模式（可选，P2）。
5. **性能基线用 pytest marker**：`@pytest.mark.perf` 标记，默认不跑（`pytest -m "not perf"`），CI 不跑（避免抖动）。手动 `pytest -m perf` 建立基线，记录到 `docs/PERF_BASELINE.md`。
6. **种子数据走 Python 脚本**：`scripts/seed_demo.py` 经 `app.run_command` 注入 5 个示例项目（含 source/version/job/script 链路），不直接写 SQL。理由：走命令总线保证一致性；可重复执行（idempotent，按 title 去重）。
7. **安装脚本走 PowerShell**：`scripts/install.ps1`（Windows）做：检查 Python 3.12+ → 创建 venv → `pip install -e .` → 创建桌面快捷方式（可选）。`scripts/uninstall.ps1` 反向。升级走 `git pull && pip install -e . --upgrade`（文档化，不写脚本）。

## 4. 决策

| # | 决策 | 理由 |
|---|---|---|
| D1 | E2E 走进程内 `run_command`，不启子进程 | CI 无 sidecar；进程内可复用 in_memory DB + fake providers；真实子进程集成推 V0.2 |
| D2 | 项目导出只导元数据，不含二进制源文件 | MVP 验收"5 个项目数据一致"看元数据；二进制文件大且涉及路径映射，推 V0.2 |
| D3 | 备份恢复走 `shutil.copy2` 文件拷贝 | 复用 bootstrap 已验证路径；SQLite 文件级备份最可靠（migrations/README:53-61 已述） |
| D4 | 日志落盘用 RotatingFileHandler（5MB×3） | 控制磁盘占用；失败降级 stderr 不阻塞启动（W8 R3 应对） |
| D5 | 性能测试用 `@pytest.mark.perf`，默认不跑 | 避免基线抖动污染常规 CI；手动跑建立基线文档 |
| D6 | 种子数据走 `run_command` 而非直写 SQL | 走命令总线保证 schema 一致性 + 可重复执行（按 title 去重） |
| D7 | 安装脚本仅 Windows PowerShell | MVP_PLAN 明确"V0.1 Windows Alpha"；macOS/Linux 推 V0.2 |
| D8 | commandType enum 一次性 +4（Layer 0 单独 PR） | 避免 3 条支线并行改 schema 文件冲突（延续 W8 D8 经验） |
| D9 | ImportProject 重映射 id（project_id + 所有 version_id） | 避免与现有项目 id 冲突；保留 `imported_from` 字段追溯 |

## 5. 风险与应对

| # | 风险 | 级别 | 应对 |
|---|---|---|---|
| R1 | E2E 测试依赖 fake providers，可能不覆盖真实 provider 失败路径 | P1 | E2E 只验证"链路通畅 + artifact 链路完整"；真实 provider 失败路径由现有单 handler 测试覆盖 |
| R2 | ExportProject 导出元数据后 ImportProject 因外键约束失败 | P0 | ImportProject 按依赖顺序插入（project → assets → versions → jobs）；重映射 id 时维护新旧 id 映射表 |
| R3 | RestoreWorkspace 覆盖 stepwork.db 时连接未关闭导致文件锁 | P0 | RestoreWorkspace 先关闭 `deps.repos.conn`，覆盖后再 `bootstrap_db` 重连；测试断言恢复后可正常查询 |
| R4 | 日志文件落盘触碰磁盘权限 | P1 | 路径走 `$STEPWORK_HOME`（已有）；FileHandler 创建失败 try/except 降级 stderr |
| R5 | 种子数据脚本重复执行产生重复项目 | P1 | 按 `title` 去重：已存在则跳过；脚本可重复执行（idempotent） |
| R6 | 性能基线在不同机器差异大 | P2 | 基线文档记录机器配置 + 允许区间；不阻塞 Gate |
| R7 | 安装脚本在无 Python 3.12 环境失败 | P1 | install.ps1 前置检查 Python 版本，缺失时给出明确错误指引 |
| R8 | ImportProject 导入恶意 zip（路径穿越/超大文件） | P0 | zip 解压用 `extractall` 前校验每个成员路径不含 `..`；限制单个 zip ≤ 50MB |
